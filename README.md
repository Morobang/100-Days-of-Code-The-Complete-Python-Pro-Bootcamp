# 🐍 Python 100 Days

> Stop vibe coding. Build the muscle memory to write Python you can actually explain.

100 days of progressive Python practice, structured for engineers who want to go deep — not just get things running.

---

## Structure

```
python-100days/
├── README.md
├── progress.md
├── setup_git.sh
├── topics/            ← cheat sheets by topic
│   ├── 01_strings.md
│   ├── 02_lists_tuples.md
│   └── ...
└── days/
    ├── day_001/
    │   ├── README.md       ← topic, objectives, connection to next day
    │   ├── notes.md        ← syntax cheat sheet, gotchas, interview tips
    │   ├── day_001.ipynb   ← your workspace
    │   └── solution.py     ← reference (read AFTER attempting)
    ├── day_002/
    └── ...
```

---

## Curriculum

| Days | Theme |
|------|-------|
| 1–10 | Foundations: strings, lists, dicts, functions, control flow |
| 11–20 | OOP: classes, inheritance, dunders, properties, patterns |
| 21–30 | Files, Errors & Modules: I/O, exceptions, stdlib |
| 31–40 | Functional Python: comprehensions, generators, decorators |
| 41–50 | DSA: recursion, Big O, sorting, linked lists |
| 51–60 | APIs & Automation: requests, scraping, scheduling |
| 61–70 | SQL & Databases: SQLite, PostgreSQL, SQLAlchemy |
| 71–80 | Data Engineering: NumPy, Pandas, ETL, PySpark |
| 81–90 | Cloud & Pipelines: boto3, S3, Lambda, Glue |
| 91–100 | Capstone Projects: end-to-end builds |

---

## Rules
1. Open `notes.md` before the notebook
2. Write your own code — no AI generation
3. Only open `solution.py` after the test cell passes (or you've been stuck 30+ min)
4. Push to GitHub every day — the streak is the accountability
